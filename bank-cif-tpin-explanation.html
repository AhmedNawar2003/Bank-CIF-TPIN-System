💡 الفكرة العامة
المشروع بيعمل إيه بالظبط؟
الـ System بيدير حسابات CIF (Customer Information File) في البنك. كل عميل عنده رقم CIF وبيحتاج يعمل TPIN (Transaction PIN) عشان يقدر يعمل عمليات.

الـ System بيحمي الـ TPIN بـ BCrypt hashing — يعني حتى لو حد وصل للـ database، مش هيلاقي الـ PIN الحقيقي. وبعد 3 محاولات فاشلة، الحساب بيتحجب تلقائياً. وبعد الـ authenticate الناجح، بيتعمل JWT token يستخدمه العميل في كل الـ requests الجاية.
PENDING_TPIN
تسجيل جديد
→
ACTIVE
بعد تحديد TPIN
→
BLOCKED
بعد 3 فشل
→
ACTIVE
بعد Reset TPIN
🗂️ الـ Layers — طبقات المشروع
المشروع بيستخدم Architecture مقسمة في طبقات واضحة
🌐
Controller Layer
controller/CifTpinController.java
بيستقبل الـ HTTP requests من المستخدم. مش بيعمل أي business logic — بس بيستقبل ويوصل للـ Service ويرجع الـ response.
⚙️
Service Layer
service/CifTpinService.java
هنا بيحصل كل الـ business logic — تسجيل، تحقق من TPIN، عد الفشل، حجب الحساب. هو "عقل" التطبيق.

Repository Layer
repository/CifAccountRepository.java
بيكلم الـ database. بيستخدم Spring Data JPA اللي بيولد الـ SQL تلقائياً من اسم الـ method.
🔐
Security Layer
security/
بيحمي الـ endpoints — كل request بيعدي عليه، بيتحقق من الـ JWT token، ولو مش موجود أو غلط بيرجع 401.

Model & DTO Layer
model/ + dto/
الـ Model هو شكل البيانات في الـ database. الـ DTOs هي شكل البيانات اللي بتيجي من المستخدم أو بترجعله.

Exception Layer
exception/
بيمسك كل الـ errors ويرجعها كـ JSON واضح بدل stacktrace. كل نوع error عنده HTTP status كود مناسب.
🗃️ الـ Data Model
ازاي البيانات بتتخزن في الـ Database
📋
CifAccount.java — الـ Entity
model/CifAccount.java
الـ @Entity دي بتقول لـ Spring إن الـ class دا جدول في الـ database. كل field بيبقى column.
الـ Fieldالنوعالهدف
cifNumberStringالرقم الفريد للعميل في البنك
fullNameStringالاسم الكامل للعميل
emailStringالإيميل (unique)
tpinHashBCrypt Hashالـ TPIN متشفر — مش plain text

failedAttemptsintعدد المحاولات الفاشلة (0→1→2→3 = blocked)
createdAt / updatedAtLocalDateTimeبيتعبئ تلقائياً بـ @PrePersist / @PreUpdate
ليه BCrypt للـ TPIN؟
BCrypt هي خوارزمية تشفير one-way — يعني مفيش طريقة ترجع منها الـ PIN الأصلي. لما المستخدم يدخل الـ TPIN، بنشفره ونقارنه بالمشفر في الـ DB.
ليه H2 In-Memory؟
H2 Database بتشتغل جوا الـ RAM — مفيش installation، مفيش setup. كل ما التطبيق بيقفل، البيانات بتتمسح. مثالية للـ development والـ testing.
🌐 الـ REST APIs — الـ 4 endpoints
كل endpoint بيعمل إيه وإيه اللي بيدخل ويطلع
POST /api/v1/cif/register Public
بيسجل CIF جديد في الـ system. الحساب بيتعمل بحالة PENDING_TPIN — يعني لازم يحدد TPIN قبل ما يقدر يعمل أي حاجة.
Request Body
{ "cifNumber": "123456789", "fullName": "Ahmed Nawar", "email": "ahmed@bank.com" }
Response — 201 Created
{ "success": true, "message": "CIF registered successfully. Please set your TPIN.", "data": { "cifNumber": "123456789", "status": "PENDING_TPIN" ... } }
POST /api/v1/cif/set-tpin Public
بيحدد TPIN للـ CIF الجديد ويغير الـ status من PENDING_TPIN لـ ACTIVE. لو الحساب ACTIVE بالفعل، بيرجع error.
Request Body
{ "cifNumber": "123456789", "tpin": "1234" // 4-6 أرقام فقط }
Response — 200 OK
{ "success": true, "message": "TPIN set successfully. Account is now active.", "data": { "status": "ACTIVE", "tpinConfigured": true ... } }
POST /api/v1/cif/authenticate Public → يرجع JWT
أهم endpoint. بيتحقق من الـ CIF والـ TPIN. لو صح بيرجع JWT token. لو غلط 3 مرات متتالية، الحساب بيتحجب.
Request Body
{ "cifNumber": "123456789", "tpin": "1234" }
Response — 200 OK
{ "success": true, "data": { "token": "eyJhbGciOiJIUzI1NiJ9...", "tokenType": "Bearer", "expiresIn": 86400 // 24 ساعة } }
POST /api/v1/cif/reset-tpin Public
بيفك الحجب عن الـ CIF المحجوب ويحدد TPIN جديد. كمان ممكن يتستخدم لو العميل عايز يغير الـ TPIN حتى لو مش محجوب.
Request Body
{ "cifNumber": "123456789", "newTpin": "5678" }
⚙️ الـ Service — عقل التطبيق
CifTpinService.java — هنا بيحصل كل الـ logic المهم
1
register() — التسجيل
بيتحقق إن الـ cifNumber مش موجود قبل كده، وإن الإيميل مش متكرر. لو الاتنين جدد، بيعمل account بحالة PENDING_TPIN.
// بيتحقق من التكرار if (cifAccountRepository.existsByCifNumber(request.getCifNumber())) { throw new CifAlreadyExistsException("CIF already registered"); } // بيبني الـ account الجديد CifAccount account = CifAccount.builder() .status(CifStatus.PENDING_TPIN) .failedAttempts(0) .build();
2
setTpin() — تحديد الـ TPIN
بيشفر الـ TPIN بـ BCrypt قبل ما يخزنه. مش بيخزن أبداً الـ plain text. بيغير الـ status لـ ACTIVE.
// التشفير — passwordEncoder هو BCryptPasswordEncoder account.setTpinHash(passwordEncoder.encode(request.getTpin())); account.setStatus(CifStatus.ACTIVE);
3
authenticate() — المصادقة
ده الـ logic الأهم — بيقارن الـ TPIN المدخول بالمشفر، وبيزود عداد الفشل، ولما يوصل لـ 3 بيحجب الحساب.
boolean tpinMatches = passwordEncoder.matches(request.getTpin(), account.getTpinHash()); if (!tpinMatches) { int newFailed = account.getFailedAttempts() + 1; account.setFailedAttempts(newFailed); if (newFailed >= 3) { // MAX_FAILED_ATTEMPTS account.setStatus(CifStatus.BLOCKED); throw new CifBlockedException(cifNumber); } throw new InvalidTpinException(3 - newFailed); // remaining } // نجاح — بيصفر العداد account.setFailedAttempts(0);
4
resetTpin() — إعادة التعيين
بيشتغل سواء الحساب محجوب أو لأ. بيشفر الـ TPIN الجديد، بيصفر العداد، ويرجع الـ status لـ ACTIVE.
account.setTpinHash(passwordEncoder.encode(request.getNewTpin())); account.setStatus(CifStatus.ACTIVE); account.setFailedAttempts(0);
🔐 الـ Security — الـ 4 Classes
كل class في الـ security package بيعمل دور مختلف
⚙️
SecurityConfig.java
بيحدد مين يقدر يدخل على أي endpoint. الـ 4 endpoints الأساسية (register, set-tpin, authenticate, reset-tpin) public. أي endpoint تاني محتاج JWT token. بيعطل الـ CSRF لأن الـ API stateless. Session نوعها STATELESS — مفيش server-side sessions.
🔑
JwtService.java
بيعمل generate للـ JWT token بعد الـ authenticate الناجح — بيحط فيه الـ cifNumber كـ subject والـ expiry كـ 24 ساعة. بيعمل validate للـ token — بيتحقق من التوقيع والـ expiry. بيعمل extract الـ cifNumber من الـ token في أي وقت.
🔎
JwtAuthenticationFilter.java
بيشتغل على كل request قبل ما يوصل للـ controller. بيقرأ الـ "Authorization: Bearer token" header. لو الـ token صح، بيعمل authentication في الـ SecurityContext. لو غلط أو مش موجود، بيعمل nothing — الـ SecurityConfig هو اللي بيحجب الـ request.
🚫
SecurityExceptionHandler.java
بيمسك الـ 401 Unauthorized (لو مفيش token أو token غلط) والـ 403 Forbidden (لو عنده token بس مش عنده صلاحية). بيرجعهم كـ JSON واضح بدل صفحة HTML من Spring.
كيف الـ JWT بيشتغل خطوة بخطوة
1
المستخدم يعمل POST /authenticate
بيبعت الـ cifNumber والـ TPIN
2
الـ Service يتحقق وينجح
الـ JwtService يعمل generate لـ token يحتوي على الـ cifNumber
3
المستخدم يستخدم الـ token
يحط الـ token في كل request: Authorization: Bearer eyJ...
4
الـ JwtAuthenticationFilter يتحقق
كل request بيعدي عليه — لو الـ token صح، الـ request بيكمل

🎬 سيناريوهات كاملة
امتى تستخدم كل endpoint
✅ السيناريو الطبيعي — Happy Path
1. التسجيل POST /register → { cifNumber, fullName, email } ← 201 | status: PENDING_TPIN 2. تحديد TPIN POST /set-tpin → { cifNumber, tpin: "1234" } ← 200 | status: ACTIVE 3. تسجيل الدخول POST /authenticate → { cifNumber, tpin: "1234" } ← 200 | token: "eyJ..." 4. استخدام الـ token GET /any-protected-endpoint Authorization: Bearer eyJ...
🔴 سيناريو الحجب — Block Scenario
POST /authenticate → tpin: "9999" ← 401 | Remaining: 2 POST /authenticate → tpin: "9999" ← 401 | Remaining: 1 POST /authenticate → tpin: "9999" ← 403 | CIF is BLOCKED — لازم يعمل reset — POST /reset-tpin → { cifNumber, newTpin: "5678" } ← 200 | status: ACTIVE (unblocked + new TPIN) POST /authenticate → tpin: "5678" ← 200 | token جديد ✓
